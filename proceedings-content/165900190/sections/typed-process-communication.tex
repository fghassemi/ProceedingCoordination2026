% \subsection{Typed Process Communication}
\label{sec:typed-process-communication}

In programming languages designed with a focus on communication, there are two main models for communication abstractions.
\emph{Channel-based} languages such as Go \cite{thegodeveloperteamGoProgrammingLanguage} and Rust \cite{matsakisRustLanguage2014}, use channels where the
sending of a message in one process is paired with the receive of the message in another process.
On the other hand, \emph{actor-based} languages such as Erlang \cite{armstrongHistoryErlang2007a} and Elixir \cite{theelixirteamElixirLanguage} operate
using \emph{mailboxes} that store messages that processes can retrieve.

% Figure \ref{fig:processes} shows the difference between both approaches in a visual way.
% Each circle represent a process, while the arrows indicate the direction of communication.

One of the most important distinctions between both models is the order in which messages are processed.
While channel-based communication imposes a strict order on the arrival of messages,
with actor-based communication, a process is typically able to choose in which order it retrieves messages from a mailbox.

% With both models there are challenges in ensuring correct communication.
% Undesired behaviour in communication protocols should be detected and avoided, such as deadlocks and protocol violations.
For both the channel-based and actor-based approach, there exist type systems that can give certain guarantees about the correctness of a protocol implementation.
