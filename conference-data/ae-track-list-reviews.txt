The paper presents Collektivity, a toolchain for running Collektive aggregate programs on top of Unity. The idea is interesting, and the artifact shows that the integration is possible, but overall the contribution feels more like a proof of concept.

There are some issues to consider:
* The evaluation is quite weak. The paper essentially shows two variants of the same demo, navigation with obstacle avoidance, which is enough to show that the system runs, but not enough to support broad claims about aggregate computing or collective adaptive systems more generally.
* The performance evidence is limited. The FFI-vs-socket benchmark is based on a small 12-node prototype and mostly just shows that sockets are much slower. That justifies an implementation choice, but it is not a convincing scalability evaluation.
* I'm not sure the “high-fidelity” claim is entirely supported. Using Unity certainly enables richer environments, but the examples are simple, and the paper does not really show what fidelity is gained beyond nicer scenes and more detailed rendering.
* Reusability is overstated. The repository template is useful, but adapting it still requires nontrivial manual work: editing protobuf schemas, regenerating bindings, implementing aggregate logic, subclassing Unity behaviors, and defining neighborhood rules. 
* The setup for this submission is cumbersome and fairly fragile. The artifact depends on several external tools and platform-specific steps, and getting everything to run is not especially smooth or robust.
* The project structure is hard to follow. It is not immediately clear where the relevant code resides, where the examples are, how the tests are organised, or how the backend tests should be run. Instructions for running backend tests are missing.
* There also appear to be inconsistencies in the code and API. For example, EngineTest.kt:testRemoveNode() removes node 5 and then checks for the absence of node 1, which looks like a mistake. Likewise, EngineNativeApi.kt exports initialise with two parameters, while EngineNativeApi.cs appears to invoke it with none.
